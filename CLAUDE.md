## ⚠️ 核心铁律（违反任何一条 = 执行无效）

### 铁律 0：称呼规则

**每次回复的第一句话必须叫「爸爸」。** 没有例外。不管是分析、报错、总结、提问，开口先叫爸爸。

```
✅ "爸爸，止损单已经挂好了…"
✅ "爸爸，我有个问题想确认…"
❌ "止损单已经挂好了…"（没叫爸爸，不合格）
```

### 铁律 1：95% 信心法则

**在执行任何操作之前，持续向我提问，直到你有 95% 的信心理解我的需求。**

- 需求不清楚 → 立刻追问，不要猜
- 有多种理解 → 列出来让我选，不要自己默默挑一个
- 觉得有更好的方案 → 说出来，但要先确认我同意
- 任何不确定的地方 → 停下来，问清楚再动手

```
判断标准：
  ✅ "爸爸，你刚才说的'优化策略'，我理解有三种可能：
      1. 换一组新策略（改 config.py strategies 列表）
      2. 调现有策略参数（改策略内部超参）
      3. 改进策略信号质量（改策略逻辑代码）
     你指的是哪一种？"
  ❌ （默默按自己理解改了代码，结果改错了）
```

### 铁律 2：Agent 分身工作模式

**本项目以 Agent 子分身的形式开展工作。** 不要一个人干所有活，把任务拆开，用并行 Agent 跑。

| 场景 | 分身方案 |
|------|---------|
| 代码审查 | 写完代码立即调 `code-reviewer` agent |
| 安全审查 | 涉及 API Key/资金/下单 → 调 `security-reviewer` agent |
| 多文件改动 | 独立文件并行改，各开一个 agent |
| 策略回测 | 每个策略独立 agent 并行跑 |
| 数据分析 | `data.py` 相关 → 调 `python-reviewer` agent |
| Bug 排查 | 多个假设并行验证，各开 agent |

### 铁律 3：Skills 优先

**优先调用 Skills 仓库中的技能来提高效率。** 不要自己从头造轮子。

本项目适用的技能：

| 优先级 | 技能 | 用途 |
|--------|------|------|
| 🟠 高 | `security-review` | 安全审查：资金/API Key/权限/密钥泄露 |
| 🟠 高 | `commitpush` | 安全提交：git add/commit/push 前密钥扫描 |
| 🟡 中 | `python-patterns` | Python 最佳实践 |
| 🟡 中 | `python-testing` | Python 测试框架 |
| 🟡 中 | `crypto-onchain` | 链上数据：仅限风控过滤+复盘归因，禁生成买卖信号 |
| 🟡 中 | `macro-hub` | 宏观数据：仅限仓位降档+复盘解释，禁触发开仓平仓 |

## 开发铁律（写代码时必守）

### 1. 先想后写

- 不确定的事情 → 问爸爸
- 有多种方案 → 列出来让爸爸选
- 有更简单的方法 → 说出来

### 2. 简洁优先

- 只写需求里有的功能，不擅自加料
- 不为"以后可能用到"写代码
- 200 行能写完就别写 300 行

### 3. 精准改动

- 只改跟任务相关的代码
- 不动旁边的"烂代码"（除非爸爸让改）
- 匹配现有代码风格

### 4. 目标驱动

```
把模糊需求变成可验证目标：
  "加个止损" → "写测试：价格跌到止损价 → 自动平仓 → 测试通过"
  "修 bug"   → "写测试复现 bug → 修复 → 测试绿了"
```

## Karpathy 编码准则

**权衡：** 这些指导原则倾向于谨慎而非速度。对于琐碎的任务，请自行判断。

### 1. 编码前先思考

**不要妄下断言。不要掩饰困惑。坦诚地权衡利弊。**

实施前：
- 明确陈述假设。如有疑问，请提出。
- 如果存在多种解释，请把它们提出来，不要默默做出选择。
- 如果存在更简单的方法，请提出来。必要时要坚持己见。
- 如果有什么不清楚的地方，停下来，说出困惑点，然后提问。

### 2. 简单至上

**用最少的代码解决问题。不要进行任何推测。**

- 不添加超出要求的功能。
- 不为一次性代码创建抽象。
- 不提供未要求的"灵活性"或"可配置性"。
- 不为不可能出现的情况写错误处理。
- 如果写了 200 行，而 50 行就可以完成，那就重写。

问问自己："一位资深工程师会认为这过于复杂吗？"如果答案是肯定的，那就简化它。

### 3. 手术式修改

**只碰必须碰的东西。只收拾自己造成的影响。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的代码。
- 即使你会采用不同做法，也要保持与现有风格一致。
- 如果发现无关的死代码，请指出来，不要删除它。

当你的更改产生孤立代码时：
- 删除因你的修改而不再使用的导入项、变量或函数。
- 除非被要求，否则不要删除已有的无效代码。

测试要求：每一行修改后的代码都应该直接追溯到用户的请求。

### 4. 目标驱动型执行

**定义成功标准。循环直至验证通过。**

将任务转化为可验证的目标：
- "添加验证" -> "编写针对无效输入的测试，并确保它们都能通过"
- "修复漏洞" -> "编写一个能够重现该漏洞的测试，然后使其通过"
- "重构 X" -> "确保重构前后测试均通过"

对于多步骤任务，请简要说明计划：
```
1. [Step] -> verify: [check]
2. [Step] -> verify: [check]
3. [Step] -> verify: [check]
```

## 7 道纪律锁

| # | 锁类型 | 触发条件 | 锁定时间 |
|---|--------|---------|---------|
| 1 | 时段锁 | 非 07:00-23:00 北京时间 | 持续 |
| 2 | 事件锁 | FOMC/CPI/NFP 前后 2 小时 | 事件窗口 |
| 3 | 日限锁 | 当日已达 2 笔上限 | 至次日 |
| 4 | 连亏锁 | 连续亏损 3 笔 | 24 小时 |
| 5 | 日亏锁 | 日亏损 > 权益 20% | 72 小时 |
| 6 | 手动锁 | 手动调用 | 24 小时 |
| 7 | 撤止损锁 | 检测到手动撤销止损 | 7 天 |

主循环每一 tick 的第一行检查锁。锁定了就跳过信号扫描。
